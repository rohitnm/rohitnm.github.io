Raleigh, NC
  • Improving OTA firmware upgrade reliability for a medical bed system, supporting increase in success rate from 90% toward 95%. // click to expand

    The system uses a dual-bank flash architecture — Bank A runs the active firmware while Bank B receives the new image. On completion, a bank-swap flag is written and the device reboots into the updated bank. The challenge: on slower eMMC parts, the write phase could exceed the watchdog window, causing the device to reset mid-swap and fall back to the old image. With no display on the unit, the only feedback was a UART log and an LED state — making field diagnosis painful.

    The fix involved restructuring the write sequence to feed the watchdog between block writes and adding a CRC checkpoint before committing the swap flag, so an interrupted update never leaves the device in a half-flashed state.

Philosophy

How I Work

📑

Think in Layers

Bare metal registers, RTOS tasks, Linux device trees, application logic — I work across the full stack. From datasheets to Yocto BSPs, I don't stay pigeonholed in one layer.

🔍

Debug Systematically

Reproduce, isolate, instrument. JTAG, logic analyzers, and printf-over-RTT are my first tools — not guessing.

✅

Test What Matters

400+ hours of automated testing at Itron taught me: catch failures early, automate the tedious, and let the system run overnight.

🛡

Ship Safe Code

Medical devices don't get second chances. MISRA C, CERT, FDA IEC 81001-5-1 — I write firmware where "it works" isn't enough; it has to be provably safe.

🔢

Visualize Before You Build

State machines, data flows, and sequence diagrams come before code. I map out firmware architecture visually — flowcharts catch design flaws that compilers can't.

📝

Document as You Go

Design docs live alongside code, not after it ships. Clean READMEs, interface specs, and decision logs — because future-you is your most important teammate.

🤖

Automate the Boring Stuff

Leverage AI for boilerplate, test scaffolding, and documentation drafts. Save brainpower for the hard problems — architecture, timing, and edge cases.

🛠

Build Tools, Not Just Firmware

When the workflow has a gap, I fill it. SVD Register Tuner and Firmware Size Profiler exist because I needed them — and so did other engineers.
