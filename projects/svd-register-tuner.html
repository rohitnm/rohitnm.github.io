The Problem

Register programming shouldn't require a PDF and a calculator.

Embedded firmware constantly touches hardware registers — GPIO, UART, SPI, DMA config. Every MCU vendor provides SVD files describing their register maps, but working with them is painful.

Reference Manual PDFs
800+ page PDFs. Find the register, find the bit field, find the offset, calculate the mask manually. Context-switch from code to PDF and back, every time.
Vendor HAL Libraries
Abstraction layers that hide the hardware. When you need to write to a specific bit, you're back to reading the register map anyway. And HALs don't cover every peripheral.
Existing SVD Viewers
Separate tool windows. No cursor awareness — you type GPIOA->ODR and still have to manually search for it. No code generation. Read-only.
Manual Bit Math
(1U << 5) — which bit was that again? Off-by-one errors in shift amounts are some of the most common embedded bugs, and they're completely silent.
Features

SVD intelligence, inside your editor.

🔍

Cursor-Driven Detection

Place your cursor on any register reference in C code — GPIOA->ODR, TIM2->CR1 — and the HUD automatically shows that register's full bit-field layout.

📊

Bit-Field Visualization

See every field: name, bit range, access type (read/write/read-only), reset value, and description. No more counting bits in hex constants.

⚡

C Code Generation

Click a bit field to generate the correct C code: set, clear, toggle, or read. Correct shift amounts and masks, every time. Paste directly into your firmware.

🗂️

Full SVD Tree Browser

Browse the complete peripheral → register → field hierarchy in VS Code's sidebar. Search by name. Jump to any register instantly.

🎯

Address & Offset Display

Shows absolute address, base + offset, and reset value for every register. Copy address to clipboard for use in debugger watch expressions.

📦

Zero Dependencies

The extension bundles its own SVD parser. No external tools, no Python, no server. Works offline, works on any OS VS Code supports.

Technical Deep Dives

The hard parts of building a register-aware IDE extension.

Cursor-to-Register Matching

C register access patterns are inconsistent: GPIOA->ODR, pGPIOA->ODR, GPIO_BASE->ODR, *(volatile uint32_t*)0x40020014. A simple regex catches the common case but misses pointer aliases and raw addresses.

Solution

Multi-strategy matcher: first tries PERIPH->REG regex on the current line, then falls back to address literal matching against the SVD address map, then checks for #define aliases. Covers ~98% of real firmware patterns.

SVD Inheritance & derivedFrom

SVD files use a derivedFrom attribute where one peripheral inherits another's register layout (e.g., GPIOB derives from GPIOA). Naive parsing misses these and shows empty peripherals.

Solution

Two-pass parse: first pass builds the base peripheral map, second pass resolves derivedFrom references by cloning the parent's register tree and patching in the child's base address. Handles nested derivation chains.

Webview Performance on Large SVDs

An STM32H7 SVD file has 80+ peripherals, 2000+ registers, 8000+ bit fields. Rendering the full tree in a VS Code Webview panel caused 500ms+ lag on every cursor move.

Solution

Lazy rendering: only the active peripheral's registers are sent to the Webview. The sidebar TreeView uses VS Code's native virtual scrolling. SVD parsing happens once at activation and is cached in memory. Cursor detection debounced to 50ms.

Bit-Field Overlap Detection

Some vendor SVD files contain overlapping bit fields (bugs in the SVD itself). Rendering these without warning produces incorrect masks and misleading visualizations.

Solution

Validation pass after parsing: checks for bit-range overlaps within each register and emits VS Code diagnostics (yellow warnings) pointing to the SVD file. The HUD highlights overlapping fields in red.

Motivation

Why this extension exists.

Every time I wrote GPIOA->MODER |= (2U << 10); I had to open the reference manual to double-check the bit offset. Every single time. That's 30 seconds of context-switching, multiplied by hundreds of register accesses per project.

The information already existed in SVD files — machine-readable, vendor-provided. No one had built a VS Code extension that made it cursor-aware and actionable. So I did.

Roadmap

Where it goes from here.

Live Debug Integration

Read register values over SWD/JTAG during debug sessions. Show live hardware state alongside the SVD description.

Multi-Field Code Gen

Select multiple bit fields and generate a single combined write. Useful for peripheral init where you configure 5+ fields at once.

Register Diff

Compare two register values (expected vs. actual) and highlight which bits differ. Invaluable for debugging peripheral init failures.

SVD Auto-Detection

Detect the target MCU from .svd files in the workspace or from launch.json. No manual file selection needed.
