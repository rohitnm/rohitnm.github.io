The Problem

Why embedded register programming is painful

📄

Datasheet Gymnastics

MCU reference manuals run thousands of pages. Finding a single register's bit layout means searching, scrolling, and cross-referencing tables — every single time, for every register.

🧮

Manual Bit Counting

Fields aren't always nicely aligned. You're counting bits, computing masks like (0x3 << 10), hoping you didn't off-by-one the shift.

🔄

Context Switching

Editor → PDF → editor → PDF → editor. Every register access breaks your flow. The context switch alone costs minutes per register interaction.

🐛

Silent Errors

Write to a read-only field? Mask the wrong bits? Forget to preserve adjacent fields with RMW? The compiler won't catch it. The hardware just misbehaves.

🔍

Raw Hex in Debug Dumps

Fault dump says 0x40021004. Which peripheral? Which register? What do the bits mean? Now decode 20 of those under pressure during a production issue.

🧑‍💻

Onboarding Time Sink

New engineer joins the team, first task: configure SPI. They spend days reading datasheets and asking seniors — mistakes a visual tool would prevent in minutes.

Features

Everything in the sidebar, nothing in the datasheet

🎯

Cursor-Driven Detection

Move your cursor to GPIOA->MODER and the HUD resolves it instantly. Supports arrow syntax, underscore style, and raw address literals.

🔲

Interactive Bit Grid

Visual 32/16/8-bit grid color-coded by field. Click any bit to toggle. Multi-bit fields use XOR toggling. Read-only bits are clearly marked.

📝

Smart Field Editors

Checkboxes for 1-bit flags, dropdown menus with actual enum names (Input/Output/Alternate/Analog), number inputs for multi-bit values.

⚡

4 Code Styles

Generate raw writes, read-modify-write, CMSIS macros, or fully commented RMW. One-click copy or insert at cursor position.

🌳

Register Browser

Collapsible peripheral tree with real-time search. Browse all 955 registers without leaving the editor. Click any register to view its fields.

🔍

Auto-Detection

Finds your target device from platformio.ini, CubeMX .ioc, Makefile defines, CMakeLists.txt, or any SVD file in the workspace. Zero configuration.

Technical Deep Dives

The hardest problems I solved

I'm an embedded engineer — C and register maps are my comfort zone, not TypeScript and CSS. The SVD domain logic (parsing phases, derivedFrom, enum propagation) was mine. For the JS/web platform bugs, I spotted the symptoms and understood the impact, then leaned on AI-assisted debugging to navigate unfamiliar language quirks and find idiomatic fixes.

XML Parser Type Coercion — 5 Bugs From One Root Cause

fast-xml-parser silently converts anything numeric-looking to a JavaScript number. SVD version "1.0" became 1. Register size "0x20" stayed a string. Descriptions that were just numbers caused .replace() is not a function crashes. This one parser behavior caused 5 separate bugs across the codebase.

→ Solution AI-assisted

Configured numberParseOptions: { hex: false, leadingZeros: false, skipLike: /[.]/ } to suppress the worst coercions. Widened cleanText() to accept string | number | undefined | null with explicit String() coercion. Added parseHexOrDec() with NaN guard for all numeric SVD fields. I knew what the SVD fields should look like — Copilot helped me find the right parser config options.

Three-Level derivedFrom Resolution

CMSIS-SVD allows peripherals, registers, AND fields to inherit from other definitions. GPIOI derives registers from GPIOA, but GPIOI appears before GPIOA in the XML. TIM5's CCR%s (dim-expanded to CCR1-CCR4) inherits from TIM2's CCR%s — requiring per-instance resolution of the %s placeholder.

→ Solution

Restructured parsing into 3 phases: (1) parse all base peripherals, (2) resolve register-level derivedFrom with dim-aware %s substitution, (3) parse derived peripherals and resolve their register inheritance. This came from reading the CMSIS-SVD spec and tracing real STM32 files by hand — pure domain work.

JavaScript 32-bit Shift Overflow

1 << 32 equals 1 in JavaScript (bitwise shifts operate modulo 32), not 4294967296. A 32-bit-wide field got a bitmask of 0x00000000 instead of 0xFFFFFFFF. This silently corrupted field extraction, bit overlap detection, and value composition — producing 14,934 false overlap reports in validation.

→ Solution AI-assisted

Created a safeMask(bitWidth) helper: bitWidth >= 32 ? 0xFFFFFFFF : (1 << bitWidth) - 1. Applied across svdParser and hudPanel. In C, 1 << 32 is undefined behavior too — but I expected it to at least produce a large number, not wrap to 1. Copilot diagnosed the JS-specific modulo-32 behavior and suggested the guard.

Enum Propagation Across Peripheral Siblings

ST's SVD defines enum values only on GPIOA (Input/Output/Alternate/Analog for MODER fields). GPIOB through GPIOI have identical field structures but no enums — showing useless 0, 1, 2, 3 dropdowns instead of meaningful names.

→ Solution

Added a post-parse propagation pass that pools all enum definitions by registerName.normalizedFieldPattern.bitWidth, then fills in missing enums on matching fields across all peripherals. I knew this was the problem immediately from working with STM32 peripheral groups — it's how ST structures their SVDs. The matching logic was straightforward once I understood the pattern.

Broken CSS Variable Concatenation

Bit highlight backgrounds used getFieldColor(ci) + '30', intending to append alpha. But getFieldColor() returned var(--field-0), producing var(--field-0)30 — invalid CSS. The bit highlight feature was completely non-functional in every build.

→ Solution AI-assisted

Switched field colors from hex values to HSL components stored as CSS custom properties. Created getFieldColorAlpha(index, alpha) that reads computed HSL at runtime and returns proper hsla(). I could see the highlight wasn't rendering — Copilot explained why CSS var() can't be string-concatenated and proposed the HSL approach, which also solved theme compatibility for free.

Motivation

Why I built this

Most embedded engineers spend half their time not writing code — they're hunting through PDFs, counting bit positions, and building register masks by hand. I've done this hundreds of times across STM32, NXP, and TI projects.

The data already exists in machine-readable form — every vendor ships CMSIS-SVD files describing every register down to the individual bit. The problem wasn't the data — it was that no tool put it where you actually work: inside the code editor, at the cursor, in real time.

SVD Register Tuner closes that gap. It's the tool I wished existed every time I typed GPIOA->MODER and then alt-tabbed to a PDF.

I used AI (GitHub Copilot) as a development accelerator — it helped with boilerplate, test scaffolding, and exploring unfamiliar VS Code APIs. The architecture, SVD parsing logic, domain decisions, and bug fixes were all mine. AI was a tool in the workflow, not the driver.
