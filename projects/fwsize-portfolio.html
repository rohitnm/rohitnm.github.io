The Problem

Flash is finite. Knowing what's using it shouldn't be hard.

On memory-constrained ARM Cortex-M devices (64 KB flash, 20 KB RAM), every byte matters. When firmware grows past the limit, you need to know what grew and why. The existing tools make this surprisingly painful.

arm-none-eabi-size
Only gives section totals: .text = 14320. No per-function breakdown. You know it grew, but not what grew.
nm --size-sort
Lists symbols with sizes, but no source file mapping. Can't answer "which module is the biggest?"
Linker .map file
10,000+ lines of raw output. Not structured, not searchable, not actionable. Nobody reads these voluntarily.
puncover / Bloaty
Unmaintained (2019) or designed for x86. No CI integration, weak Cortex-M support, no diff mode.
Features

One tool. Complete size intelligence.

📊

Per-Function Breakdown

See exactly how many bytes each function and variable contributes to flash and RAM. Sorted, grouped by source file, with percentages.

🔍

DWARF Source Mapping

Maps every symbol to its source file using DWARF debug info. Uses a ±16B window scan to handle compiler alignment — the same technique as GDB.

⚖️

Binary Diff

Compare two ELF builds side-by-side. Shows added, removed, and changed symbols with byte deltas. Catch regressions before they ship.

💀

Dead Code Detection

Finds unreferenced functions via relocation analysis. Catches indirect references (function pointers, ISR vectors) that call-graph analysis misses.

🚦

CI Quality Gate

Set flash/RAM limits. Exit code 1 on failure. Drop fwsize --check into any CI pipeline and catch size regressions automatically.

🗺️

Interactive HTML Report

Squarified treemap, donut pie charts, sortable tables. Self-contained single HTML file — no CDN, works offline.

📁

Multiple Output Formats

Terminal (Rich color tables), HTML (treemap), JSON (machine-readable), CSV (spreadsheet). Use --format to pick.

⚡

Instant Results

Parses a real Cortex-M3 ELF in under 50ms. No build system integration needed — just point it at any .elf file.

Technical Deep Dives

The hard problems behind the clean output.

Building a correct firmware profiler requires understanding the full embedded toolchain — from compiler output to linker layout to debug info encoding.

DWARF Address Misalignment

~30% of functions initially showed <unknown> source file even with -g. The compiler inserts alignment NOPs before function prologues, so the symbol's start address doesn't exactly match any DWARF line entry.

Solution

Implemented a ±16 byte window scan around each symbol address. If the exact address isn't in the DWARF map, check nearby bytes. This is the same technique GDB and addr2line use internally. Source mapping went from ~70% to >95%.

The .data Double-Counting Trap

Most size tools count .data as either flash or RAM. In reality, initialized global variables live in both: initial values are stored in flash at _sidata, and the startup code copies them to RAM. Counting only one underestimates actual usage.

Solution

Created a "flash+ram" classification. The aggregator counts .data symbols toward both flash_used and ram_used, giving accurate utilization percentages that match what the hardware actually consumes.

Dead Code via Relocations (Not Call Graphs)

Naive dead code detection traces the call graph. But in embedded firmware, many functions are invoked indirectly: ISR handlers through the vector table, callbacks via function pointers, __attribute__((weak)) aliases. Call-graph analysis misses all of these.

Solution

Parse relocation sections (.rel.* / .rela.*). Any symbol that appears in a relocation entry — whether from a direct call, a function pointer store, or a vector table entry — is marked as referenced. Combined with an allow-list for known entry points (Reset_Handler, ISR vectors). Zero false positives in testing.

Squarified Treemap Visualization

A flat bar chart makes small symbols invisible. The HTML report needed a 2D treemap where area = bytes, so engineers can visually spot the biggest consumers at a glance.

Solution

Implemented a squarified treemap algorithm in vanilla JavaScript — recursive slice-and-dice with aspect ratio optimization. Each block is proportional to its byte count, colored by source file, labeled when large enough. Self-contained single HTML file, no external libraries, works offline.

AI-assisted: Copilot accelerated the treemap layout algorithm and SVG pie charts. Reviewed and tuned manually.
Test Firmware

Tested against real compiled firmware. Not mocks.

To validate fwsize against real-world binaries, I wrote a complete bare-metal Cortex-M3 firmware from scratch — 7 source files, compiled with arm-none-eabi-gcc, targeting the TI LM3S6965 (64 KB flash, 20 KB RAM).

test_firmware/ 
test_firmware/
├── startup.c        # ISR vector table, Reset_Handler, .data copy, .bss zero
├── main.c           # Application entry — init peripherals, run scheduler
├── uart.c           # UART driver — send byte, send string, receive
├── gpio.c           # GPIO driver — pin mode, read, write, toggle
├── spi.c            # SPI driver — init, transfer, chip select
├── scheduler.c      # Cooperative round-robin task scheduler
├── ring_buffer.c    # Lock-free ring buffer for UART RX
├── analytics.c      # v2 only — exercises the diff feature
├── link.ld          # Linker script: FLASH 0x0 (64K), RAM 0x20000000 (20K)
└── Makefile         # Build both demo.elf and demo_v2.elf

Two ELF variants are built: demo.elf (base firmware) and demo_v2.elf (adds analytics module). This exercises the diff, dead code, and size comparison features with real linker behavior, real DWARF debug info, and real section layouts.

Motivation

Why this project exists.

On every embedded team I've worked on, the same question comes up at least once a sprint: "The firmware doesn't fit anymore — what grew?"

The answer is always a 30-minute detective session with nm, size, and a linker map file. fwsize reduces that to a single command. I built the tool I wished I had.

It's also a demonstration of how embedded engineering and software engineering intersect: binary format parsing, compiler toolchain knowledge, CI/CD integration, and clean CLI design — all in one project.

Roadmap

Where it goes from here.

--watch Mode

Auto re-runs when the ELF changes. Save → build → instant size report in your terminal. No manual re-invocation.

RISC-V & ESP32 Targets

Add architecture-specific section classifiers for RISC-V and Xtensa. Most of the codebase is already arch-agnostic — ~20 lines of changes.

Linker .map Fallback

Parse .map files for stripped ELFs without .symtab. Enables analysis of release builds without debug info.

GitHub Actions PR Comments

A published action that posts size diff comments on pull requests — like Codecov does for coverage. Track firmware size trends over time.

PyPI Publishing

Publish to PyPI so anyone can pip install fwsize and use it instantly. Package config is already in place via pyproject.toml.

VS Code Extension

Inline CodeLens annotations showing flash/RAM cost per function. Right-click context menu for size details. A natural next step once the CLI is published on PyPI.

--blame Mode

Cross-reference symbol addresses with Git history. Answer "who added this 2 KB function and when?" in one command.
