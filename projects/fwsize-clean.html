Flash is finite. Knowing what's using it shouldn't be hard.

On memory-constrained ARM Cortex-M devices (64 KB flash, 20 KB RAM), every byte matters. When firmware grows past the limit, you need to know what grew and why. The existing tools make this surprisingly painful.

arm-none-eabi-size

Only gives section totals: .text = 14320. No per-function breakdown. You know it grew, but not what grew.

nm --size-sort

Lists symbols with sizes, but no source file mapping. Can't answer "which module is the biggest?"

Linker .map file

10,000+ lines of raw output. Not structured, not searchable, not actionable.

puncover / Bloaty

Unmaintained (2019) or designed for x86. No CI integration, weak Cortex-M support, no diff mode.

One tool. Complete size intelligence.

Per-Function Breakdown

See exactly how many bytes each function and variable contributes to flash and RAM. Sorted, grouped by source file, with percentages.

DWARF Source Mapping

Maps every symbol to its source file using DWARF debug info. Uses a ±16B window scan to handle compiler alignment — the same technique as GDB.

Binary Diff

Compare two ELF builds side-by-side. Shows added, removed, and changed symbols with byte deltas. Catch regressions before they ship.

Dead Code Detection

Finds unreferenced functions via relocation analysis. Catches indirect references (function pointers, ISR vectors) that call-graph analysis misses.

CI Quality Gate

Set flash/RAM limits. Exit code 1 on failure. Drop fwsize --check into any CI pipeline and catch size regressions automatically.

Interactive HTML Report

Squarified treemap, donut charts, sortable tables. Self-contained single HTML file, works offline.

Multiple Output Formats

Terminal (Rich color tables), HTML (treemap), JSON (machine-readable), CSV (spreadsheet). Use --format to pick.

Instant Results

Parses a real Cortex-M3 ELF in under 50ms. No build system integration needed — just point it at any .elf file.

The hard problems behind the clean output.

Building a correct firmware profiler requires understanding the full embedded toolchain — from compiler output to linker layout to debug info encoding.

DWARF Address Misalignment

~30% of functions initially showed <unknown> source file even with -g. The compiler inserts alignment NOPs before function prologues, so the symbol's start address doesn't exactly match any DWARF line entry.

Solution

Implemented a ±16 byte window scan around each symbol address. If the exact address isn't in the DWARF map, check nearby bytes. Same technique GDB and addr2line use internally. Source mapping went from ~70% to >95%.

The .data Double-Counting Trap

Most size tools count .data as either flash or RAM. In reality, initialized globals live in both: initial values stored in flash at _sidata, startup code copies them to RAM.

Solution

Created a "flash+ram" classification. The aggregator counts .data symbols toward both flash_used and ram_used, giving accurate utilization percentages that match what the hardware actually consumes.

Dead Code via Relocations (Not Call Graphs)

Naive dead code detection traces the call graph. But in embedded firmware, many functions are invoked indirectly: ISR handlers through the vector table, callbacks via function pointers, __attribute__((weak)) aliases.

Solution

Parse relocation sections (.rel.* / .rela.*). Any symbol appearing in a relocation entry is marked referenced. Combined with an allow-list for known entry points. Zero false positives in testing.

Squarified Treemap Visualization

A flat bar chart makes small symbols invisible. The HTML report needed a 2D treemap where area = bytes, so engineers can visually spot the biggest consumers at a glance.

Solution

Squarified treemap algorithm in vanilla JavaScript — recursive slice-and-dice with aspect ratio optimization. Each block proportional to byte count, colored by source file. Self-contained single HTML file, no CDN.

AI-assisted: Copilot accelerated the treemap layout algorithm and SVG pie charts. Reviewed and tuned manually.

Why this project exists.

On every embedded team I've worked on, the same question comes up at least once a sprint: "The firmware doesn't fit anymore — what grew?"

The answer is always a 30-minute detective session with nm, size, and a linker map file. fwsize reduces that to a single command. I built the tool I wished I had.

It's also a demonstration of how embedded engineering and software engineering intersect: binary format parsing, compiler toolchain knowledge, CI/CD integration, and clean CLI design — all in one project.

Where it goes from here.

--watch Mode

Auto re-runs when the ELF changes. Save → build → instant size report in your terminal.

RISC-V & ESP32

Architecture-specific section classifiers. Most of the codebase is already arch-agnostic — ~20 lines of changes.

Linker .map Fallback

Parse .map files for stripped ELFs without .symtab. Enables analysis of release builds.

GitHub Actions PR Comments

Posts size diff comments on pull requests — like Codecov does for coverage. Track firmware size trends over time.

PyPI Publishing

pip install fwsize and use it instantly. Package config already in place via pyproject.toml.

--blame Mode

Cross-reference symbol addresses with Git history. Answer "who added this 2 KB function and when?"
